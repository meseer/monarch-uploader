# WebView Authentication & Token Capture — Low-Level Design

> **Status:** Draft  
> **Updated:** 2026-02-25  
> **Author:** @meseer  
> **Note:** Low-level design for the WebView login flow and token/cookie capture mechanism. Part of the [Mobile App Architecture](./architecture.md).

---

## 1. Overview

Each financial institution integration requires the user to log in before the app can make API calls on their behalf. The login itself happens in a standard embedded WebView — the user sees and interacts with the institution's real login page. The app's role is to capture the resulting session credentials (tokens or cookies) from the HTTP traffic generated by that login, then store them securely for use in subsequent native API calls.

Two interception strategies are used depending on what needs to be captured:
- **Request interception** — captures outgoing `Authorization: Bearer <token>` headers set by the web app after login
- **Response interception** — captures `Set-Cookie` headers in login responses for cookie-based sessions

Both operate at the **HTTP network layer**, below JavaScript. This is more reliable than reading `localStorage` or `sessionStorage` via injected JS because it does not depend on the internal structure of the web app.

## 2. Platform Interception Mechanisms

### 2.1 iOS — `WKURLSchemeHandler` and `WKNavigationDelegate`

iOS provides two complementary hooks for WebView network access:

**`WKURLSchemeHandler`** handles custom URL schemes but cannot intercept `https://` directly. The standard approach for HTTPS interception uses a local proxy or a `URLProtocol` subclass registered before the `WKWebView` is instantiated.

**Practical approach for React Native:** Use `react-native-webview`'s `onShouldStartLoadWithRequest` combined with a **JavaScript injection** that patches `XMLHttpRequest` and `fetch` to post auth headers back to the native layer via `window.ReactNativeWebView.postMessage`. This is the most cross-platform-compatible approach within the Expo managed workflow.

For cases requiring true HTTP-layer interception (where JS injection is unreliable), a thin native module using `WKURLSchemeHandler` with a custom scheme redirect can be written as an Expo native module.

### 2.2 Android — `WebViewClient.shouldInterceptRequest`

Android's `WebViewClient` provides `shouldInterceptRequest(WebView view, WebResourceRequest request)` which fires for **every network request** made by the WebView, including XHR and fetch calls. This gives access to request URLs and headers.

For response headers (needed for `Set-Cookie` capture), `shouldInterceptRequest` can return a modified response, or an OkHttp-based approach can be used via a custom `WebViewClient` subclass.

### 2.3 JavaScript Injection (Cross-Platform Fallback)

For integrations where tokens are accessible in the web app's JavaScript runtime, injecting a script that reads from `localStorage`, `sessionStorage`, or patches `fetch`/`XHR` is a simpler cross-platform alternative. This is the primary approach for the Expo managed workflow without custom native modules.

**Injection pattern:**
```javascript
// Injected into WebView after login page loads
(function() {
  // Patch fetch to intercept Authorization headers
  const originalFetch = window.fetch;
  window.fetch = function(url, options) {
    const auth = options?.headers?.Authorization || options?.headers?.authorization;
    if (auth && auth.startsWith('Bearer ')) {
      window.ReactNativeWebView.postMessage(JSON.stringify({
        type: 'AUTH_TOKEN_CAPTURED',
        token: auth.replace('Bearer ', ''),
        url: url
      }));
    }
    return originalFetch.apply(this, arguments);
  };
  
  // Patch XMLHttpRequest for older API clients
  const originalOpen = XMLHttpRequest.prototype.open;
  const originalSetRequestHeader = XMLHttpRequest.prototype.setRequestHeader;
  XMLHttpRequest.prototype.setRequestHeader = function(name, value) {
    if (name.toLowerCase() === 'authorization' && value.startsWith('Bearer ')) {
      window.ReactNativeWebView.postMessage(JSON.stringify({
        type: 'AUTH_TOKEN_CAPTURED',
        token: value.replace('Bearer ', ''),
      }));
    }
    return originalSetRequestHeader.apply(this, arguments);
  };
})();
```

## 3. Per-Integration Token Capture Strategy

### 3.1 Questrade

**What to capture:** Questrade's web app uses multiple tokens with different scopes. The current userscript reads these from the page's session/local storage after login.

**Capture method:** After the user completes login, inject JS to read from `localStorage` and `sessionStorage` for known Questrade token keys. Post all found token entries to native code via `ReactNativeWebView.postMessage`.

**Token rotation:** Questrade rotates tokens on use. The app must save the new token returned in each API response and discard the old one.

**Multiple tokens:** Different capabilities (accounts, balances, positions, activities) may use different token scopes. The injection should capture all token keys matching Questrade's known patterns.

```javascript
// Injected after login completion detected
const tokens = {};
['localStorage', 'sessionStorage'].forEach(storageType => {
  const storage = window[storageType];
  for (let i = 0; i < storage.length; i++) {
    const key = storage.key(i);
    // Capture all Questrade-related keys
    if (key && (key.includes('questrade') || key.includes('iq') || key.includes('token'))) {
      tokens[key] = storage.getItem(key);
    }
  }
});
window.ReactNativeWebView.postMessage(JSON.stringify({
  type: 'QUESTRADE_TOKENS',
  tokens
}));
```

> **Note:** The exact storage keys for Questrade tokens need to be determined by inspecting the Questrade web app's network traffic. This is a known gap to be filled during implementation.

### 3.2 Wealthsimple

**What to capture:** A single Bearer token sent in the `Authorization` header of GraphQL requests to `my.wealthsimple.com/graphql`.

**Capture method:** XHR/fetch patch injection. Watch for requests to `my.wealthsimple.com/graphql` and extract the `Authorization` header.

**Token lifetime:** Wealthsimple tokens have a known expiry. The token response includes `expires_in`. Store the expiry timestamp alongside the token.

**Silent re-login:** If stored credentials (email + password) are available in Keychain, perform a silent re-login by submitting the login form programmatically in a hidden WebView. If 2FA is triggered, surface the WebView to the user.

### 3.3 Rogers Bank

**What to capture:** A Bearer token sent in the `Authorization` header of requests to `online.rogersbank.com/EBPP/api/v1`.

**Capture method:** XHR/fetch patch injection. Watch for requests to the Rogers Bank API base URL.

**Additional identifiers:** The Rogers Bank integration also requires `accountId`, `customerId`, and `deviceId` which are extracted from API responses. The response interception should also capture these from the first successful API response after login.

**Silent re-login:** Rogers Bank may tie sessions to device fingerprints. Silent re-login should replay the same device headers captured during the original WebView login.

### 3.4 Canada Life

**What to capture:** Session cookie for `my.canadalife.com` (Salesforce/Aura-based session). Specifically the `$AuraClientService.token$siteforce:communityApp` token stored in the page's JavaScript context.

**Capture method:** JS injection to read the Aura client service token from `localStorage` after the dashboard loads. This specific key is already known from the userscript (`STORAGE.CANADALIFE_TOKEN_KEY`).

```javascript
// Injected after dashboard URL detected
const token = localStorage.getItem('$AuraClientService.token$siteforce:communityApp');
if (token) {
  window.ReactNativeWebView.postMessage(JSON.stringify({
    type: 'CANADALIFE_TOKEN',
    token
  }));
}
```

**Session duration:** Canada Life sessions are relatively short-lived. Silent re-login may need to happen frequently. Stored credentials enable this without user interaction unless MFA is required.

### 3.5 MBNA

**What to capture:** Session cookie for `service.mbna.ca`. The cookie is set by the login response and sent automatically with subsequent requests.

**Capture method:** After successful login (detected by URL navigation to the accounts page), use `react-native-cookies` / `CookieManager` to extract all cookies for the `service.mbna.ca` domain from the WebView's cookie jar.

```javascript
// Native code (React Native) — runs after WebView navigates to accounts page
import CookieManager from '@react-native-cookies/cookies';

const cookies = await CookieManager.get('https://service.mbna.ca');
// Store relevant session cookie in expo-secure-store
```

**Note:** MBNA cookies are HttpOnly in most cases, meaning they cannot be read via `document.cookie` from injected JS. The `CookieManager` approach reads directly from the WebView's native cookie store, which does have access to HttpOnly cookies.

## 4. Login Completion Detection

The app needs to know when login is complete to trigger token capture. Detection strategies per integration:

| Integration | Completion Signal |
|---|---|
| Questrade | URL navigates to `/app/` or `/accounts` |
| Wealthsimple | URL navigates to `my.wealthsimple.com/app/` dashboard |
| Rogers Bank | URL navigates to `online.rogersbank.com` accounts page |
| Canada Life | URL navigates to `my.canadalife.com/.../dashboard` |
| MBNA | URL navigates to `service.mbna.ca/` accounts overview |

In `react-native-webview`, use `onNavigationStateChange` to watch for these URL patterns and trigger the appropriate extraction logic.

## 5. Secure Token Storage Schema

All tokens and credentials are stored in `expo-secure-store` (Keychain on iOS, Keystore on Android). Keys are namespaced by integration:

```
Keychain key                              Value (JSON string)
─────────────────────────────────────────────────────────────
integration:questrade:tokens              { key1: "...", key2: "..." }
integration:questrade:credentials         { username: "...", password: "..." }
integration:wealthsimple:token            { token: "...", expiresAt: 1234567890 }
integration:wealthsimple:credentials      { email: "...", password: "..." }
integration:rogersbank:token              { token: "...", accountId: "...", ... }
integration:rogersbank:credentials        { username: "...", password: "..." }
integration:canadalife:token              { token: "..." }
integration:canadalife:credentials        { username: "...", password: "..." }
integration:mbna:cookies                  { sessionId: "...", ... }
integration:mbna:credentials              { username: "...", password: "..." }
app:monarchToken                          { token: "..." }
```

**Credential storage is opt-in.** The user must explicitly choose to save credentials for background sync. The UI makes it clear that credentials are stored in the device's secure enclave.

## 6. Silent Re-Login Flow

When a token is expired and stored credentials are available:

```
1. Create a hidden (off-screen) WebView
2. Navigate to the institution's login page
3. Wait for the login form to render
4. Inject JS to fill in username and password fields
5. Submit the form
6. Watch for:
   a. Success → login completion URL detected → capture new token → dismiss hidden WebView
   b. 2FA challenge → surface the WebView to the user, send push notification if app is backgrounded
   c. CAPTCHA or bot detection → surface WebView to user
   d. Error → mark integration as needing re-authentication, notify user
```

**Security note:** Credential auto-fill via JS injection into a WebView is only done against the institution's own domain. The app validates that the WebView URL matches the expected institution domain before injecting credentials.

## 7. Network Allowlist Check

Before any sync operation (foreground or background), the app checks the current network:

```javascript
import NetInfo from '@react-native-community/netinfo';

async function isNetworkAllowed(allowlist) {
  const state = await NetInfo.fetch();
  
  if (state.type === 'wifi') {
    // Check SSID against allowlisted wifi networks
    const ssid = state.details?.ssid;
    return allowlist.wifi.includes(ssid);
  }
  
  if (state.type === 'cellular') {
    // Cellular is allowed by default if it's in the list
    return allowlist.cellular === true;
  }
  
  // VPN, ethernet, unknown — not allowed by default
  return false;
}
```

The allowlist is configured by the user in settings and synced to the backend for multi-device consistency.

## 8. Open Questions

1. **Questrade token key names** — the exact `localStorage`/`sessionStorage` keys used by the Questrade web app need to be determined by inspecting the app during a live session. This should be done as part of the Phase 1 implementation spike.

2. **MBNA HttpOnly cookies** — `CookieManager` has access to HttpOnly cookies from native code, but this needs to be verified on both iOS and Android during implementation.

3. **App Store review risk** — injecting JavaScript into third-party banking WebViews to extract credentials is a sensitive capability. The implementation should be reviewed against Apple's App Store Review Guidelines (particularly 5.1.1 — Data Collection and Storage) before submission. Positioning this as a personal-use tool for the user's own accounts mitigates risk significantly.

4. **Wealthsimple token format** — the token structure (JWT vs. opaque) and rotation policy should be confirmed during the implementation spike.
